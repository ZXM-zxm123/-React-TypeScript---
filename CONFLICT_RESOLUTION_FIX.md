# 协作电子表格 - 冲突解决修复说明

## 问题描述

原实现存在严重的并发编辑问题：当多人同时编辑相同单元格时，后提交的内容会无条件覆盖先提交的内容，没有任何冲突检测或处理机制。

## 解决方案概述

我们实现了基于单元格粒度的版本控制和冲突检测机制，主要包含：

### 1. 服务端改进 (Node.js)

- **单元格版本跟踪**: 每次单元格更新都会增加版本号
- **冲突检测**: 在处理单元格更新时，检查版本号是否匹配
- **待处理编辑管理**: 将冲突的编辑保存起来
- **冲突解决**: 支持三种解决方式：
  - 保留服务器当前值
  - 保留客户端提交的值
  - 使用自定义合并值

### 2. 前端改进 (React)

- **本地版本跟踪**: 维护每个单元格的本地版本号
- **编辑带版本**: 每次更新发送当前期望的版本号
- **冲突解决 UI**: 提供美观的弹窗让用户选择如何处理冲突
- **三种解决选项**:
  - 保留服务器值
  - 保留我的编辑
  - 自定义合并

### 3. 类型扩展 (TypeScript)

- 添加了版本和时间戳字段到 CellData 接口
- 添加了 PendingEdit 和 CellConflict 接口
- 完善了类型安全

## 使用方式

### 冲突发生时

当系统检测到编辑冲突时，会弹出冲突解决对话框，提供三个选项：

1. **保留服务器值** - 保留服务器端当前的值，放弃你的编辑
2. **保留我的编辑** - 使用你的编辑覆盖服务器端的当前值
3. **自定义合并** - 输入一个合并后的值，将其应用为最终值

### 防止冲突的最佳实践

- 建议不同用户专注在电子表格的不同区域
- 可以使用颜色来标识当前正在编辑的单元格（已有功能）
- 如果发现另一个用户光标在某个单元格上，等待几秒再编辑该单元格

## 未来改进方向

虽然当前的冲突解决已经能很好处理基本场景，但可以进一步优化：

1. **操作转换 (OT)**: 实现更复杂的自动合并算法
2. **CRDT 数据结构**: 使用更现代的分布式数据结构
3. **冲突历史**: 记录所有发生的冲突历史供后续查看
4. **实时锁定**: 在检测到多个用户准备编辑相同单元格时进行实时锁定提示

## 运行说明

### 启动 Java 服务
```bash
cd java-service
mvn spring-boot:run
```

### 启动 Node.js 服务
```bash
cd server
npm install
npm run dev
```

### 启动前端开发服务器
```bash
cd frontend
npm install
npm run dev
```

打开两个浏览器窗口连接到同一个房间，尝试同时编辑相同的单元格即可体验冲突解决功能！
